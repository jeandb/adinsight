# Story 02.4: Cards de métricas adicionais de pedidos com comparação ao período anterior

**Status:** Done
**Epic:** Revenue Dashboard
**Traces to:** PRD Seção 8 — Integração WooCommerce / Dashboard de Faturamento

## User Story

Como gestora de tráfego ou diretora, quero ver cards com métricas consolidadas adicionais sobre os pedidos do período (Receita Líquida, Taxa de conclusão e Clientes únicos), cada um com a variação em relação ao período anterior, para entender a saúde do negócio e identificar tendências rapidamente.

## Acceptance Criteria

- [x] 3 novos cards aparecem na seção de pedidos: **Receita Líquida**, **Taxa de conclusão**, **Clientes únicos**
- [x] **Receita Líquida** = `total_cents` de pedidos `completed` − `total_cents` de pedidos `refunded`
- [x] **Taxa de conclusão** = `completed` ÷ (`total` − `processing`) × 100, exibida em %
- [x] **Clientes únicos** = `COUNT(DISTINCT customer_email)` de todos os pedidos do período
- [x] Cada card exibe abaixo do valor principal a variação (Δ) em relação ao período anterior equivalente, no formato `+X%` (verde) ou `−X%` (vermelho)
- [x] O período anterior é calculado com a mesma duração do período ativo, imediatamente antes dele (mesma lógica já usada nos KPIs de receita)
- [x] Os cards respeitam o filtro de período ativo (`after`/`before`)
- [x] Os dados vêm de um endpoint dedicado `GET /revenue/orders-metrics` (separado do `orders-summary` da Story 02.2)
- [x] Loading state: 3 skeletons enquanto carrega
- [x] Se não houver pedidos no período, os cards exibem zero/`—` sem quebrar

## Technical Notes

- Novo endpoint backend: `GET /revenue/orders-metrics?after=&before=`
- O backend calcula `after`/`before` do período anterior a partir dos parâmetros recebidos:
  ```
  periodDays = ceil((before - after) / 86400000)
  prevBefore = after
  prevAfter  = after - periodDays * 86400000
  ```
- Query SQL sugerida (período atual e anterior em uma única query com CASE WHEN):
  ```sql
  SELECT
    COUNT(DISTINCT CASE WHEN order_date >= $1 AND order_date <= $2 THEN customer_email END) AS unique_customers,
    COUNT(DISTINCT CASE WHEN order_date >= $3 AND order_date <  $4 THEN customer_email END) AS prev_unique_customers,
    COUNT(CASE WHEN order_date >= $1 AND order_date <= $2 THEN 1 END)                       AS total_orders,
    COUNT(CASE WHEN order_date >= $3 AND order_date <  $4 THEN 1 END)                       AS prev_total_orders,
    COUNT(CASE WHEN order_date >= $1 AND order_date <= $2 AND status = 'completed'   THEN 1 END) AS completed,
    COUNT(CASE WHEN order_date >= $3 AND order_date <  $4 AND status = 'completed'   THEN 1 END) AS prev_completed,
    COUNT(CASE WHEN order_date >= $1 AND order_date <= $2 AND status = 'processing'  THEN 1 END) AS processing,
    COUNT(CASE WHEN order_date >= $3 AND order_date <  $4 AND status = 'processing'  THEN 1 END) AS prev_processing,
    COALESCE(SUM(CASE WHEN order_date >= $1 AND order_date <= $2 AND status = 'completed' THEN total_cents END), 0) AS completed_cents,
    COALESCE(SUM(CASE WHEN order_date >= $3 AND order_date <  $4 AND status = 'completed' THEN total_cents END), 0) AS prev_completed_cents,
    COALESCE(SUM(CASE WHEN order_date >= $1 AND order_date <= $2 AND status = 'refunded'  THEN total_cents END), 0) AS refunded_cents,
    COALESCE(SUM(CASE WHEN order_date >= $3 AND order_date <  $4 AND status = 'refunded'  THEN total_cents END), 0) AS prev_refunded_cents
  FROM woo_orders
  WHERE order_date >= $3 AND order_date <= $2
  ```
- Frontend: novo `useQuery` com `queryKey: ['revenue', 'orders-metrics', period, dateFrom, dateTo]`
- Reutilizar o componente `KpiCard` já existente na `RevenuePage` para exibição consistente
- Posicionar os 3 novos cards **abaixo** dos 4 cards de status (Story 02.2)
